# DCSync Detection in Active Directory using Splunk

## Overview

This project simulates a DCSync attack in a controlled Active Directory lab and focuses on how the activity can be detected and investigated using Splunk.

The goal here was not just to execute the technique, but to understand how abnormal replication behaviour appears in logs, how to identify it, and how basic remediation would be handled from a SOC perspective.

This was performed in a lab environment, so while the behaviour reflects real-world techniques, actual enterprise environments would involve more noise, stricter controls, and more complex investigations.

---

## Lab Setup

- Domain Controller (Windows Server)
- Windows 11 (Domain-joined)
- Kali Linux (attacker machine)
- Splunk (running on host)
- Sysmon enabled

---

## Attack Simulation

A DCSync attack was simulated from the Kali machine using Impacket with compromised domain user credentials (`HRHindocha`).

The attack was possible because replication privileges ("Replicating Directory Changes") were incorrectly assigned to a non-domain controller account.

Using this, the attacker was able to impersonate a domain controller and request password hash data from the domain.

## Initial Detection (Event ID 4662)

The first step in detection was identifying directory service access events in Splunk.

Query used:


index=* EventCode=4662
| table _time, Account_Name, Logon_ID, Properties, IpAddress
| sort _time

## Investigation – Behaviour Analysis

To validate whether this activity was abnormal, the events were grouped based on properties.

Query used:


index=* EventCode=4662 Account_Name="HRHindocha"
| stats count by Properties, EventCode


<p align="center">
  <img src="../../screenshots/AD Screenshots/DCSync/Query2.png" width="800"/>
</p>

<p align="center">
  <img src="../../screenshots/AD Screenshots/DCSync/Query2Result.png" width="800"/>
</p>

The results showed a high number of **Control Access** operations, which is a strong indicator of directory replication activity.

---

## Validation – Log Inspection

A deeper look into a single event showed:

- **Access Mask:** `0x100`
- **Properties:** Control Access GUID
- **Account:** Non-DC user (`HRHindocha`)

<p align="center">
  <img src="../../screenshots/AD Screenshots/DCSync/FinalEvidence.png" width="800"/>
</p>

This confirms that the account was performing replication actions typically reserved for domain controllers.

---

## Source Correlation

To understand how the activity originated, the Logon ID was correlated with logon events.

Query used:


index=* EventCode=4624 Logon_ID=0x99A59F
| table _time, host, user, Source_Network_Address, Logon_Type


<p align="center">
  <img src="../../screenshots/AD Screenshots/DCSync/Query3.png" width="800"/>
</p>

<p align="center">
  <img src="../../screenshots/AD Screenshots/DCSync/Query3Result.png" width="800"/>
</p>

The result showed:

- **Logon Type 3 (Network Logon)**
- Source IP corresponding to the Kali attacker machine

This confirms that the activity was initiated remotely rather than locally on the domain controller.

---

## Conclusion of Detection

At this stage, the following was established:

- A non-domain controller account (`HRHindocha`) performed replication actions
- A large number of Event ID 4662 logs were generated
- Control Access operations indicated directory replication behaviour
- Logon Type 3 confirmed a remote attack source

This aligns with DCSync attack behaviour.

## Key Takeaways

- Event ID **4662** is critical for detecting directory replication abuse
- DCSync detection relies on identifying **who is performing replication**, not just where from
- **Control Access + replication GUIDs** are strong indicators
- Logon correlation (4624) helps identify attack origin
- Misconfigured permissions are often the root cause of high-impact attacks

---

## Final Note

This project was completed in a controlled lab to understand how DCSync behaviour appears in logs and how it can be investigated.

In real environments, this type of activity would be more difficult to isolate due to noise and legitimate replication traffic, but the same principles apply.

The focus here was on building a clear understanding of detection and response rather than just executing the attack.
